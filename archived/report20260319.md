# AI-QUIC 工作汇报

2026-03-19

## 1. 项目简介

AI-QUIC 是一个面向学习、验证和分阶段演进的 QUIC 协议实验仓库。项目目标不是一次性做出完整产品，而是按 QUIC 协议栈的关键能力逐层推进，把底层协议构件、连接管理、恢复控制、迁移能力和应用层接入能力逐步落地，并在每个阶段都配套验证入口。


项目当前已从早期的“协议部件验证”推进到“具备稳定 transport/recovery 基础、连接管理能力、应用层接入能力和基础观测能力”的阶段。

## 2. 阶段划分

本项目按阶段推进，每一阶段都对应一类核心能力。

### 2.1 阶段 0：连接基础骨架

目标是先建立统一连接对象和统一运行框架，为后续 TLS、收发、流和恢复打基础。重点包括：

- 连接状态机
- 包号空间
- 统一收包入口
- 统一发包计划入口
- 定时器与事件入口

### 2.2 阶段 1：TLS 1.3 与 QUIC 握手核心

目标是把 BoringSSL QUIC 接口接入进来，建立真实握手路径。重点包括：

- TLS 1.3 QUIC 回调层
- CRYPTO 数据重组
- transport parameters 注入与解析
- Handshake 和 1-RTT 密钥安装
- 最小真实握手闭环

### 2.3 阶段 2：完整包收发流程

目标是把握手之外的实际收发流程补齐，使连接能在真实 UDP 条件下运行。重点包括：

- Initial、Handshake、1-RTT 的真实收发
- ACK 生成
- Version Negotiation 与 Retry
- anti-amplification 限制
- example 级真实 UDP 验证

### 2.4 阶段 3：流、多流与流控

目标是从“连接能建立”推进到“连接能承载业务数据”。重点包括：

- stream 状态机
- 发送缓冲与接收重组
- FIN / RESET / STOP_SENDING
- MAX_DATA / MAX_STREAM_DATA / MAX_STREAMS
- 多流双向数据传输

### 2.5 阶段 4：恢复与拥塞控制

目标是补齐 RFC 9002 的核心能力，让系统在丢包、延迟和大流量条件下具备稳定性。重点包括：

- RTT 估计
- loss detection
- PTO
- 拥塞窗口演化
- bulk 传输与阻塞回归
- 拓扑 profile 下的恢复验证

### 2.6 阶段 5：连接管理与迁移

目标是把 QUIC 从“能传输”推进到“能处理路径变化、关闭和异常情况”。重点包括：

- CID 生命周期
- path validation
- preferred-address 迁移
- idle timeout
- closing / draining
- stateless reset 基础路径

### 2.7 阶段 6：应用层接入、观测与互操作收尾

目标是把前面几阶段形成的 transport 能力提升成可对外接入、可调试、可验证的系统。阶段 6 又可以分成两个方向：

- 子阶段 A：稳定应用 API、app demo、事件、metrics、基础 fuzzing
- 子阶段 B：外部 interop、系统化 fuzzing、长期 benchmark，以及后续 HTTP/3/QPACK 扩展

当前项目处于阶段 6，且子阶段 A 已完成。


## 3. 整体设计架构

本轮工作的一个重点，是把仓库逐步整理成清晰的分层结构，而不是继续在单个示例或单个测试里堆功能。

### 3.1 协议基础层

这一层负责最底层的协议构件，包括：

- 包头预解析
- 版本分派
- Initial 密钥派生
- 包保护与解保护
- transport parameters
- 帧语法解析

这一层的目标，是保证 QUIC 的基础报文和协议元素具备独立可验证性，为上层连接和握手逻辑提供稳定底座。

### 3.2 连接与传输控制层

这一层承接 QUIC 的核心运行语义，包括：

- 统一连接对象
- 包号空间管理
- 握手驱动与密钥生命周期
- ACK 与在途包管理
- 流状态机与流控
- 恢复与拥塞控制
- 路径、迁移、关闭语义

这一层是整个项目的主体，也是本轮阶段推进的主战场。阶段 0 到阶段 5 的主要成果都集中在这里。

### 3.3 应用接入与观测层

在前述 transport/recovery 基础上，阶段 6 开始明确抽出应用层接口，不再让示例程序直接绑定内部实现细节。

这一层的目标包括：

- 提供稳定的应用层接入 API
- 暴露连接、路径、stream 的只读状态
- 输出事件、指标与基础日志
- 支撑最小 app demo
- 为后续 interop 和上层协议扩展留接口

这标志着项目开始从“协议实现”转向“可使用、可调试、可验证的系统”。

### 3.4 验证与工程支撑层

除了协议代码本身，本轮还持续建设了一层独立的工程支撑体系，用来保证长期可迭代：

- `README.md` 作为全局状态面板
- `API.md` 作为对外接口说明
- `Makefile` 作为统一构建与测试入口
- `topo.py` 作为真实网络验证脚本
- `tests/` 作为阶段化验证矩阵
- `log`、`debug`、`break` 作为过程沉淀与调试恢复机制
- 本地 `skill` 作为方法论和约束的固化载体

这一层的意义，是让项目后续继续推进时，不再依赖“记忆中的上下文”，而是依赖可持续复用的文档、流程和工具。

## 4. 工作推进方式

本轮工作的另一个重点，是形成了比较清晰的阶段推进方法，而不是单次补丁式开发。

### 4.1 按阶段边界推进

事先要求codex在readme中对每个阶段进行详细计划，包括需要完成的功能，测试等

所有工作都尽量回到 `README.md` 中的阶段完成标志来判断范围：

- 阶段 0 解决连接骨架问题
- 阶段 1 解决 TLS/QUIC 握手问题
- 阶段 2 解决完整包收发问题
- 阶段 3 解决流和流控问题
- 阶段 4 解决恢复与拥塞控制问题
- 阶段 5 解决连接管理与迁移问题
- 阶段 6 解决应用接入、观测、fuzzing 与 interop 收尾问题

这样做的好处是，开发目标和验证目标始终一致，不容易出现“代码做了很多，但阶段定义并没有闭环”的问题。

### 4.2 按 RFC 约束推进

本轮继续坚持一条原则：凡是涉及 QUIC 协议行为的修改，优先对照本地 `rfc-docs/` 和相关 RFC 文本。

### 4.3 按三层验证推进

所有重要变更都尽量沿着三层验证链路走：

1. 分阶段测试
2. 本机 loopback 示例验证
3. `topo.py` 真实拓扑验证

其中前两步由codex自动执行，第三步由于权限问题交由用户操作，在了解测试代码功能的基础上，可直观检查测试结果，并交由codex分析。

### 4.4 skill的使用

除了代码编写规范，以及上述几点之外，skill中还包含以下指导任务流程的语句：

1. 每进行一步较大的操作，向`log`中打印记录
2. 自动测试-修复bug时，在`debug`文件中记录原因，修复过程，结果等，供用户检查和回溯
3. 需要临时中断代码生成时，将已完成和未完成任务分离，记录在`break`文件中，供重启时参考
4. 自动总结用户常用，但未添加到skill中的要求


## 5. 代表性问题与处理方式

从整体上看，本轮遇到的问题主要分为四类：

- 握手与密钥生命周期问题
- 真实网络场景下的收发与退出时序问题
- 流、ACK、恢复相关的跨层耦合问题
- 阶段 5 迁移语义与路径切换问题

这些问题的共同特点是：

- 不是单个函数层面的错误，而是跨越协议语义、状态机、example 行为和测试方法的组合问题
- 单靠单元测试不一定能一次暴露
- 需要通过 `debug`、`log`、增强输出和分层回归共同收敛

因此，本轮最大的调试成果并不只是“修好了若干 bug”，而是形成了一套能处理复杂协议耦合问题的工作方式。

（划分大阶段粒度过于粗糙，直接交由codex执行容易出现较大的耦合bug。再对每个阶段进一步生成计划，可以减少这种情况的发生）

## 6. 当前工作进度

### 6.1 已完成部分

当前已经完成代码编写并通过测试的部分包括：

- 阶段 0 到阶段 4 的主体能力
- 阶段 5 的核心连接管理与迁移路径
- 阶段 6 子阶段 A 的应用 API、基础观测、app demo 和 fuzz smoke

### 6.2 正在推进部分

当前仍在推进中的主要内容是阶段 6 的后半部分，包括：

- 外部实现 interop
- 更系统的 fuzzing 体系
- 更长期的 benchmark 框架

下一步的首个外部 interop 对象，当前选择 `xquic`。


- `quic-interop-runner`
  - 负责 correctness 和 feature coverage
- `quic-network-simulator`
  - 负责 transport benchmark 和长期性能基线

后续 interop 建议按以下顺序推进：

1. 先做本机 loopback 的双实现 smoke test
2. 再接入标准化 interop case
3. 稳定后再扩大到更重的 loss、migration 和多连接场景
4. 等 transport 与应用接口更稳定后，再进入 HTTP/3 相关互通


### 6.3 尚未进入实现闭环的部分

后续仍属于中长期扩展项的包括：

- HTTP/3 / QPACK
- 更完整的 0-RTT / resumption / ticket 语义
- Key Update
- 更全面的跨实现 benchmark 与 feature matrix
