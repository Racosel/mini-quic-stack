# 大型代码库审查器

## 概览

审查大型仓库时不要试图一次读完全部内容。先快速建立系统地图，聚焦真正相关的执行路径，依据证据排序风险，再交付能够解决问题的最小安全代码与文档变更。

## 快速开始

1. 用一句话明确目标：评审、审计、修复缺陷、新增功能、重构，或补齐文档。
2. 在仓库根目录确认 `log` 文件存在；如果不存在，先创建。
3. 在仓库根目录确认 `debug` 文件存在；如果不存在，先创建。
4. 每完成一个执行步骤，都向根目录 `log` 追加记录，至少写明时间、步骤、涉及文件或命令，以及结果。
5. 每追加一条日志前，都重新获取当前时间；不要复用任务开始时或上一条日志的时间戳。`log` 的时间格式固定为 `[%Y-%m-%d %H:%M:%S %z]`，前后使用英文中括号。
6. 每当发现 bug、确认根因、完成修复或补齐验证时，都向根目录 `debug` 追加 Markdown 记录，至少写明时间、阶段、现象、影响、定位结论、修复方法和验证结果。
7. 每次收到用户贴回的运行结果，或自己从自动脚本中拿到输出时，都要先检查 `debug` 中相关 bug 的状态：确认解决后标记为“已解决”；如果同一个 bug 是经过后续调整才彻底修好，就在原有条目里继续补现象、分析、修复和验证，不默认新开替代 bug。
8. 如果怀疑代码行为违反某个 IETF/RFC 规范，必须先查本地 `rfc-docs/` 或官方 RFC 文本，确认规范要求后再修改代码，不能凭记忆修正协议行为。
9. 在打开大量文件之前，先检查仓库整体结构。
10. 只追踪控制当前行为所需的最小文件集合。
11. 编辑前先写清假设、风险和验证路径。
12. 以最小范围实现改动，并在同一轮中同步更新测试和文档。
13. 代码重构或整理时，在必要位置保留适量的逻辑描述、关键函数签名和有助于理解行为边界的说明，不要把关键语义信息一并删除。
14. 对本仓库的测试相关改动，遵循项目约定：测试产物放在 `tests/` 目录下，自动化验证默认重复执行 5 轮，`make clean` 需要清除除 `boringssl/build/` 外的全部二进制和临时测试文件。
15. 对本仓库的 `Bug 008` 类跨层耦合缺陷，要把最终结论沉淀为高信号摘要，不要把 `debug` 长期维护成纯时间线流水账；执行细节放在 `log`。
16. 若修改 `src/tls/quic_tls.c`、`src/transport/quic_stream.c`、`src/recovery/quic_ack.c`、`example/client.c`、`example/server.c`、`topo.py` 或阶段 12-14 相关测试，默认视为高回归风险变更，至少回归 `make test12 test13 test14`、本机文件传输回环和 `make topo-auto-file`。
17. 当用户贴回本机测试或自动化脚本的失败输出，而当前日志不足以定位问题时，可以在 `example/`、测试或自动化脚本的失败/超时路径中临时补充高信号调试打印，优先输出状态机阶段、路径状态、关键标志位、待发送/待验证状态和约束条件，不要无节制增加低价值噪声。
18. 任务推进过程中要及时更新 `README.md` 中与当前任务直接相关的内容，至少同步任务边界、完成度、验证方式和测试进度；不要把 README 只当作最终收尾文档。

当仓库足够大、`rg --files` 输出噪声过多时，优先使用 `scripts/repo_inventory.py`。

## 执行日志要求

- 每一个执行步骤都必须更新仓库根目录的 `log` 文件。
- 如果 `log` 不存在，先创建，再继续后续操作。
- 仓库根目录的 `debug` 文件专门用于记录执行过程中遇到的 bug、定位结论、修复方法和验证结果；如果不存在，先创建。
- 记录应包含时间、步骤说明、涉及的文件或命令，以及结果、结论或下一步。
- 每追加一条日志前，都必须重新获取一次当前时间，不能复用任务开始时或上一条日志的时间戳。
- `log` 统一使用 `date '+%Y-%m-%d %H:%M:%S %z'` 获取时间，并以 `[时间] 内容` 形式写入。
- 对同一任务中的连续小步骤，可以合并成一条高信号日志，但不能完全省略。
- `log` 和 `debug` 分工不同：前者记步骤，后者记缺陷与修复，不能混用替代。
- `debug` 应使用 Markdown 语法，优先包含时间、阶段、标题、现象、影响、根因、修复、验证、相关文件和状态。
- 每次拿到新的运行输出后，要先回看 `debug` 并维护状态：已确认解决的 bug 标记为“已解决”，同一 bug 的后续分析与最终修复继续追加到原条目。
- 当某个 bug 已演化为“缺陷群”并跨越多轮修复后，要主动把条目重构成“主现象 / 主根因 / 修复面 / 验证 / 后续风险”的摘要结构，避免后续阅读成本失控。

## 工作流

### 1. 明确任务边界

- 明确用户要求的结果、可能涉及的子系统，以及成功标准。
- 优先检查本地约束：`AGENTS.md`、项目文档、lint/test 配置和贡献规则。
- 如果任务是评审，先输出问题发现，不要先写总结。
- 如果任务是修复缺陷或新增功能，先定义哪些稳定行为不能被破坏。

### 2. 建立高信号地图

- 先用 `rg --files`、`rg -n` 和 `git status --short` 这类快速结构化命令。
- 当你需要快速掌握语言分布、最大文件和高密度目录时，使用 `scripts/repo_inventory.py <repo-root>`。
- 在深入实现细节之前，先读入口文件、路由或初始化文件、构建清单和测试。
- 与其扫描整个目录，不如每个子系统先读少量有代表性的文件。
- 在心中持续维护以下判断：
  - 请求或控制流如何经过系统
  - 状态边界和数据归属在哪里
  - 模块之间通过什么接口协作
  - 你将要改动的路径附近有哪些测试覆盖

### 3. 先审计，再编辑

- 在提出代码改动前，先审视正确性、安全性、性能、迁移和运维风险。
- 对怀疑的问题，要用代码路径、调用点、测试或配置作为证据。
- 对受标准约束的协议行为，还要补上对应 RFC/IETF 条款作为修改依据。
- 明确区分以下三类情况：
  - 已确认缺陷
  - 高概率缺陷
  - 代码异味或可维护性风险
- 需要严重级别模型或审计清单时，阅读 `references/audit-rubric.md`。

### 4. 规划变更边界

- 优先选择在正确层级上修复问题的最小闭环补丁。
- 除非现有模式本身就是缺陷来源，否则尽量保持既有风格。
- 在大型代码库中，不要顺手重写无关部分，除非用户明确要求。
- 如果变更跨越模块边界，编辑前先列出会被触及的文件和契约。
- 如果仓库本身已存在未提交变更，要隔离用户修改与自己的修改，不要回退无关内容。

### 5. 安全实现

- 除非仓库约定必须提交生成产物，否则只修改源文件，不改生成文件。
- 在行为变化附近补充或调整测试。
- 新增代码时，只添加解释非显然控制流、不变量或安全假设所必需的注释。
- 修复缺陷时，优先在代码或测试中证明不变量，而不是只写文档说明。
- 文档变更要和行为变更同步：视情况更新 README、API 文档、配置文档、迁移说明或架构说明。
- 当用户贴回失败输出且信息不足时，可以先增强失败/超时打印，再进入下一轮定位；调试输出要服务于缩小状态空间，而不是堆砌日志。
- `README.md` 不只是介绍文档，也是项目当前任务与测试进度的高层状态面板；阶段推进时要持续维护。

### 6. 验证并收尾

- 先跑最窄且最相关的检查，再视需要扩大验证范围。
- 如果完整验证成本过高，要明确说明哪些已运行、哪些未运行，以及剩余风险。
- 做评审时，按严重级别、文件位置、影响和触发条件报告问题。
- 做实现任务时，概述行为变化、验证结果和剩余后续项。

## 决策规则

- 对于仓库本身可以回答的问题，先搜索和检查结构，再决定是否提问。
- 要主动压缩上下文。浅读五十个文件，通常不如深读五个文件。
- 相比猜测，更依赖代码、测试、模式定义、配置和局部 diff 这类一手证据。
- 当代码行为、配置或运维预期变化时，把文档视作交付物的一部分。
- 如果改动可能影响数据完整性、鉴权、兼容性、迁移或公共 API，要明确升级风险提示。

## 评审输出格式

对于审计或评审请求，按以下顺序组织输出：

1. 按严重级别排序的问题发现，并附文件引用。
2. 影响结论可信度的开放问题或前提假设。
3. 仅在确有帮助时补充简短的变更摘要。

如果没有发现问题，要明确写出来，并说明剩余风险或测试空白。

## 文档更新要求

- 当标志位、接口、CLI 行为、配置或安装步骤变化时，更新面向用户的文档。
- 当架构、不变量、扩展点或运维流程变化时，更新面向开发者的文档。
- 当新代码引入新的集成模式或使用方式时，补充示例。
- 当任务涉及功能开发、重构、迁移或 API 变化时，阅读 `references/documentation-playbook.md`。

- `references/audit-rubric.md`：提供大型项目评审所需的严重级别模型、证据规则和审计清单。
