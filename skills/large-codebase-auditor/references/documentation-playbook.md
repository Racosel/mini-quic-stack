# 文档更新手册

## 目录

1. 需要更新什么
2. 不同变更类型的最小文档要求
3. 编写规则

## 需要更新什么

### 行为变化

- 当可观察输出、API 响应、CLI 行为或 UI 契约发生变化时，更新面向用户的文档。
- 如果新行为容易产生歧义，补充示例。

### 配置变化

- 记录新增的标志位、环境变量、配置项、默认值和失败模式。
- 说明该变更是否向后兼容，以及省略配置值时会发生什么。

### 接口变化

- 更新 API 文档、请求与响应示例、类型定义和迁移说明。
- 如有必要，说明弃用窗口和兼容性限制。

### 运维变化

- 当安装、部署、调试或恢复步骤变化时，更新运行手册或开发者文档。
- 对迁移和有状态变更补充发布与回滚说明。

## 不同变更类型的最小文档要求

### 缺陷修复

- 描述用户可见症状，或现在已被强制保证的不变量。
- 仅当上线后仍然相关时，再说明运维风险。

### 新功能

- 说明如何使用、如何配置，以及适用约束。
- 至少补充一个贴近真实场景的示例。

### 重构

- 仅在外部行为、扩展点或维护流程发生变化时更新文档。
- 如果公共行为没有变化，优先简化代码而不是增加解释性文字。

### 迁移

- 说明前置条件、执行顺序、回滚预期和兼容性边界。
- 对单向数据变化要明确说明。

## 编写规则

- 让文档尽量靠近它解释的代码。
- 优先使用具体示例，而不是泛泛描述。
- 当仓库里已经有权威示例时，不要在 prose 中重复大段代码。
- 如果改动引入了不明显的不变量，优先在测试中编码，其次才是写注释。
